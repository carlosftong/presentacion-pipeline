Punto de partida

La pregunta que cambió cómo construyo software

Al principio la pregunta era "¿cómo le pido esto al modelo?"

Con el tiempo esa pregunta fue cambiando hacia "¿cómo entrego software que funciona?"

Esa diferencia — del prompt al SDLC — es lo que fue dando forma a todo lo que construí.

De dónde venía y hacia dónde fue evolucionando

El punto de partida habitual cuando se empieza a construir con IA es el prompt: "¿cómo le pido esto al modelo?" No es un mal punto de partida — es el natural. El problema aparece cuando se queda ahí.

Lo que fui descubriendo es que el modelo es una pieza dentro de un proceso más grande: diseño → código → tests → seguridad → despliegue. Si el proceso no está definido, el agente ejecuta bien dentro de un caos bien automatizado.

La automatización amplifica lo que ya existe. Si el proceso tiene baches, la IA los recorre más rápido. Por eso el SDLC pasó a ser el punto de partida, no el prompt.

Lo que fui entendiendo

Tres capas que fueron tomando forma

🗺️
Estratégico — ¿Qué construir y por qué?
Visión del producto, casos de uso, épicas de trabajo. Sin esta capa, los agentes ejecutan sin dirección.
🏗️
Técnico — ¿Cómo arquitectarlo?
SDLC como pipeline, arquitectura hexagonal, patrones de agentes. Define las reglas del juego.
⚙️
Operacional — Claude Code ejecuta el plan
Sesiones con nombre y objetivo. Agentes especializados. Artefactos trazables. El plan se vuelve código.
Cómo se fueron diferenciando estas tres capas

No surgieron juntas. Al principio todo era operacional: abrir Claude, pedir código, iterar. Funcionaba para cosas pequeñas. Cuando los proyectos crecieron, empezaron a aparecer problemas difíciles de depurar: decisiones técnicas sin criterio claro, trabajo que no conectaba con el objetivo real.

Lo que fui aprendiendo es que estas tres capas son distintas y necesitan atención separada. Mezclarlas genera ruido; separarlas da claridad:

Estratégico sin técnico = visión sin arquitectura. Técnico sin estratégico = ingeniería sin propósito. Operacional sin las dos = automatización del caos.

Claude Code vive en la capa operacional — y funciona mejor cuando las otras dos ya tienen respuestas.

Algo que costó aprender

Las sesiones sin objetivo consumen sin avanzar

Abrir Claude Code sin un objetivo claro se siente productivo. Pero al final de la sesión es difícil saber qué se logró. Nombrar la sesión y definir el objetivo antes de abrirla cambió eso.

Sesión de ideación

Pensar antes de hacer

Espacio para explorar, ajustar épicas y definir dirección. Sin presión de entrega.

Sesión de ejecución

Hacer con propósito

Arranca con la épica del proyecto y su carpeta. Contexto completo desde el minuto uno.

Qué pasaba antes y qué cambió

Las primeras sesiones eran abiertas: "trabajemos en el proyecto". El resultado era inconsistente — a veces mucho avance, a veces horas de trabajo que no quedaban claras en el historial. Nombrar cada sesión — por ejemplo sesion/car-39-hexagonal-foundation — fue un cambio simple con impacto claro:

  • Obliga a aclarar el objetivo antes de empezar, no durante
  • Crea un registro entendible del historial de trabajo
  • Define el scope del agente: qué puede y qué no puede tocar

La sesión de ideación es distinta: no tiene entregable, solo exploración. Todo lo que emerge ahí alimenta las sesiones de ejecución que siguen.

Qué cambió concretamente después de esa sesión

Antes de esa sesión, los proyectos tenían vida propia pero no conversaban entre sí. No había jerarquía clara, ni forma de saber en qué punto estaba cada uno al retomar trabajo. La GenIA Strategy definió 5 épicas, cada una con su proyecto Linear, su carpeta local y sus historias:

  • Kitsune Studio — Pipeline multiagente de producción de anime
  • HardCode 30x — Pagos verificados entre agentes IA
  • Pipeline Multiagente — SDLC autónomo como pipeline
  • Arquitecto de Agentes — Sistema de recomendación de arquitecturas
  • Fundamentos de Ingeniería — Estándares transversales a todos

Después de esa sesión, cada vez que se abre un proyecto existe: la épica activa, las historias pendientes y la carpeta con todo el contexto. El costo de retomar bajó significativamente.

Las malas prácticas de Kitsune que informaron todo lo demás

Kitsune fue el primer experimento real con agentes en producción. Funcionó — pero dejó ver varios problemas que no eran obvios al principio:

  • Sin recuperación: si algo fallaba, había que reiniciar el pipeline completo desde cero
  • Sin gates humanos: los agentes ejecutaban fases completas sin ningún punto de revisión
  • Sin aislamiento: todos los agentes compartían el mismo contexto y el mismo espacio de archivos
  • Criterios implícitos: el éxito de una fase era "que se vea bien", no algo medible
  • Estado manual: el seguimiento dependía de un archivo status.md que nadie actualizaba de forma consistente

Ninguno de estos problemas era evidente al diseñar Kitsune. Aparecieron al operar. Cada uno generó una decisión de diseño en el Pipeline Multiagente.

Cuando el issue de Linear se convierte en input del pipeline, el seguimiento de proyecto y la ejecución de código comparten el mismo lenguaje.

Cómo surgió esta conexión y qué resolvió

Antes de esta integración, el seguimiento en Linear y el trabajo en Claude vivían separados. Había que actualizar ambos manualmente y era fácil que se desincronizaran. La pregunta que lo resolvió fue: ¿qué pasa si el issue de Linear es también el input del pipeline?

  • El seguimiento ve CAR-001 como "en progreso" en Linear
  • El agente recibe CAR-001 con su spec y criterios de aceptación
  • Los artefactos generados referencian CAR-001 en nombre y metadata
  • El PR cierra CAR-001 al hacer merge — sin pasos manuales

No se trata de dos sistemas sincronizados — es el mismo sistema visto desde dos ángulos. Eso elimina una categoría entera de errores de sincronización.

Qué aprendí al dejar de usar Claude como asistente genérico

El patrón que no funcionaba bien: un solo prompt largo pidiendo diseño + código + tests juntos. El resultado era inconsistente y difícil de depurar. Lo que funcionó fue separar responsabilidades como lo haría un equipo real.

Cada agente solo lee lo que necesita, solo escribe en su espacio y solo conoce su fase. Eso redujo interferencias y hizo los resultados más predecibles:

  • Test y Security corren en paralelo — sin esperar uno al otro
  • Cada agente tiene criterios de aceptación medibles, no solo "que funcione"
  • Los gates G1, G2, G3 son los puntos donde el humano decide — el agente no aprueba su propio trabajo

Pensar en roles y contratos — en lugar de en prompts — fue lo que hizo al sistema más robusto.

Épica 3 — AI-DLC

Un sistema que automatiza el ciclo de desarrollo

El Pipeline Multiagente recibe un PRD como input y produce código desplegado como output, pasando por diseño, desarrollo, tests, seguridad y despliegue — con humanos en los puntos donde las decisiones importan.

🔁
Dos zonas de autonomía
Zona 1: humano + Claude co-crean el diseño. Zona 2: agentes ejecutan autónomamente con gates de aprobación.
🤖
6 agentes especializados
Architecture · Functional Design · Dev · Test · Security · Deploy — cada uno con su fase, su worktree y su contrato.
🔌
Agnóstico al ejecutor LLM
Funciona con Claude Code y OpenCode. Comunicación via filesystem + git. Sin dependencias de runtime específico.
Las cuatro premisas que guiaron su diseño

El diseño del pipeline no partió de preferencias técnicas — partió de cuatro restricciones que surgieron de los problemas encontrados en proyectos anteriores:

  • Agnóstico: el sistema no puede depender de un modelo específico. Si cambia el proveedor, el pipeline sigue funcionando
  • Eficiente: cada agente recibe solo los artefactos que necesita para su fase. Los prompts son templates con placeholders, no dumps de contexto completo
  • Trazable: log JSONL append-only por fase, status.md regenerado en cada transición. Cualquier decisión de cualquier agente puede auditarse
  • Human-in-the-loop: el humano guía la ideación y especificación. Los gates G1, G2 y G3 son puntos de aprobación que no se pueden saltear

Estas premisas convirtieron al pipeline en algo más que un experimento: son las condiciones mínimas para que sea confiable en producción.

Qué puede hacer el pipeline hoy y qué sigue

El pipeline puede ejecutar el ciclo completo de forma end-to-end: recibe un PRD, genera diseño, produce código, lo testea y lo despliega — con los tres gates humanos en los puntos correctos.

Algunas características que se fueron sumando a lo largo de las waves:

  • Architecture Agent: una fase previa a Dev que genera el diseño hexagonal del componente antes de escribir código
  • Inner loops por agente: Dev, Test, Security y Deploy tienen su propio ciclo interno con granularidad de tarea, no de fase completa
  • Workspace isolation: los artefactos no se inyectan en los prompts — cada agente lee directamente desde su worktree
  • Model routing: Sonnet para orquestación, Haiku para exploración, sub-agentes paralelos donde aplica

Activo hoy: incorporar principios SOLID en los prompts de Dev Agent y Architecture Agent (CAR-125 y CAR-126). El pipeline sigue evolucionando en cada sesión de trabajo.

Lo que quedó después de todo el proceso

Un SDLC que se puede ejecutar, trazar y recuperar

Ejecutable

Cada fase es un agente

Con inputs definidos, outputs estructurados y criterios de éxito medibles.

Trazable

Todo queda registrado

JSONL append-only, artefactos por fase, Linear como fuente de verdad del ciclo de vida.

Recuperable

Sin rehacer trabajo

Si una fase falla, se reintenta esa fase — no el pipeline completo.

Humanos donde importa

Gates en puntos críticos

G1, G2, G3 son decisiones humanas. Los agentes exploran; las personas aprueban.

Por qué estas cuatro propiedades no eran evidentes al principio

En Kitsune, ninguna de estas cuatro existía. Los pipelines funcionaban en sesiones controladas pero fallaban al escalar o al retomarse días después. Cada una de estas propiedades vino de un problema real:

  • Ejecutable vino del problema de Kitsune: reiniciar todo al fallar una fase. Ahora cada fase es independiente
  • Trazable vino del problema del estado manual: un status.md que nadie mantenía. Ahora el JSONL es append-only y automático
  • Recuperable vino de perder trabajo al fallar: worktrees aislados + max retries por fase sin afectar las anteriores

Estas propiedades no se diseñaron en papel — emergieron de operar sistemas que no las tenían. Esa es la única razón por la que funcionan en la práctica.
